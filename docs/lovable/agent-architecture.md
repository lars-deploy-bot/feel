# Agent Architecture (How Code Gets Written)

The most architecturally significant investigation area — and the one with the biggest remaining unknowns.

## Claude Code is NOT Running Inside the Sandbox

No Claude Code process was found running. The sandbox binary (`tini -- /home/api/sandbox`) is the only long-running process (PID 1). Between user messages, the working tree is always clean — no background agent process modifying files.

## `lovable-exec` — Dev Toolchain, NOT an AI Agent

**The single most important clarification.** Initially assumed to be the AI agent orchestrator. It is not. It is a **task runner** for dev server lifecycle management.

**Identity**: Go binary (`github.com/lovable/sandbox-cli`), compiled from:
```
/build/cmd/lovable-exec/main.go
/build/internal/taskrunner/taskrunner.go
/build/internal/taskrunner/tagger.go
/build/internal/taskrunner/vite.go
/build/internal/taskrunner/vite_command.go
```

**Usage**:
```
Usage: lovable-exec [flags] <task> [-- extra-args...]

Tasks: install, dev, build, build:dev, test, lint, start

Flags:
  -dry-run              Print command without executing
  -feature-vite-override Enable Vite config override for legacy projects
  -w string             Working directory
  -work-dir string      Working directory
```

### `lovable.toml` Full Schema (reverse-engineered)

Parsed by structs `taskrunner.Config`, `DevServerConfig`, `ViteConfig`:

```toml
[run]
# Override commands for each task. Highest priority in resolution chain:
# lovable.toml → Justfile → package.json scripts
dev = "bun run dev"
build = "bun run build"
install = "bun install"
test = "vitest run"
lint = "eslint ."
start = "bun run start"

[dev_server]
port = 8080           # Override the dev server port

[paths]
# Working directory or output path overrides (field names unknown)

[vite]
override = true       # Enable/disable the auto-generated vite config override
                      # (feature-flagged as "tagger-vite-override.enabled")
```

The file is optional. Error message: `no command found for task %q. To fix this, add to lovable.toml: [run] %s = "your-command-here"`.

### Vite Config Override Template (full)

```typescript
// Auto-generated by lovable-exec - DO NOT EDIT
import { defineConfig } from "vite";
{{.ImportLine}}
import { componentTagger } from "lovable-tagger";

const userConfig = {{.ConfigExpr}};
const lovableTagger = componentTagger();

export default defineConfig(async (configEnv) => {
  const resolvedConfig =
    typeof userConfig === "function" ? await userConfig(configEnv) : userConfig;
  const rawPlugins = resolvedConfig?.plugins;
  const plugins = Array.isArray(rawPlugins) ? rawPlugins : rawPlugins ? [rawPlugins] : [];
  const existingPlugins = plugins.filter((p) => p?.name !== "lovable-tagger");
  // injects lovable-tagger into plugins, deduplicating if already present
});
```

Go `text/template` wraps user's Vite config and injects lovable-tagger + lovite without touching user's original file. `{{.ImportLine}}` is the user's original config import, `{{.ConfigExpr}}` is the user's config value expression. Handles both function and object configs.

**Bottom line**: `lovable-exec` is the equivalent of our `bun run dev` / `bun run build` wrapper. It is NOT the thing that decides what code to write.

## The Missing Piece: Agent Invocation

**We know what lovable-exec does (dev toolchain). We know what the sandbox binary does (infrastructure). How does the AI agent get invoked?**

Possible architectures:
1. **Claude Code spawned via gRPC `StartRequest`** — platform sends `claude --message "..."` to sandbox, Claude Code runs, writes files, exits.
2. **Direct API calls from outside** — platform calls Anthropic's API directly, sends file diffs to sandbox via `apply-patch` or direct writes. Single-file commits by `gpt-engineer-app[bot]` support this.
3. **Hybrid** — Claude Code inside for complex tasks, direct API + patches for simpler edits.

## Inferred Architecture

```
User message → Platform API → ???
  → [agent invocation mechanism unknown]
    → writes files to /dev-server/ (via apply-patch or direct writes)

```

## Imported Codebase Flow

When `LOVABLE_IMPORTED_CODEBASE` env var is set, an alternate pipeline activates:
- Uses `devenv`-managed dev server instead of standard Vite
- Checks out `.envrc` instead of standard setup (with `--force`, tolerates missing file)
- Skips normal install/build — "deployment: skipping for imported codebase (devenv manages dev server)"
- On dependency changes: "imported codebase - devenv manages dependencies"
- Restarts dev server after checkout: "deployment: restarting dev server for imported codebase"

This is how non-standard projects (not Vite templates) work — `devenv`/`direnv` manages the entire environment.
